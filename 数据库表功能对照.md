# 数据库表功能对照表（blog-springboot）

> 最后更新：2026-07-22
> 适用范围：Spring Boot 3.2.5 + JPA（ddl-auto: update）+ MySQL
> 现状：库里共 **18 张表 = 12 张活跃表 + 6 张孤儿表**

---

## 一、12 张活跃表（代码在用，JPA 自动维护）

按模块归类，每个主表对应一个 CRUD 模块；关系/支撑表依附主表，属有意为之的共享设计，非耦合违规。

| 表名 | 对应功能 | 所属模块 | 说明 |
|------|---------|---------|------|
| `t_user` | 用户账户：注册 / 登录 / JWT / 个人信息 / 头像关联 | 用户 | 主表，自带完整 CRUD |
| `t_article` | 博客文章：CRUD + 发布/下线状态流转 | 文章 | 主表，自带完整 CRUD |
| `t_favorite` | 收藏的文章（用户↔文章，`user_id+article_id` 唯一约束） | 文章·收藏 | 关系表，依附 `t_user`/`t_article` |
| `t_like` | 点赞的文章（用户↔文章，唯一约束） | 文章·点赞 | 关系表，依附 `t_user`/`t_article` |
| `t_announcement` | 站内公告 | 公告 | 主表，自带完整 CRUD |
| `t_announcement_read` | 公告已读标记（用户↔公告） | 公告 | 关系表，依附 `t_user`/`t_announcement` |
| `t_relic` | 线上博物馆文物：上传 / 审核（待审/已驳回/已发布） | 博物馆 | 主表，自带完整 CRUD |
| `t_product` | 古董铺闲置商品：上架 / 在售 / 已售 / 下架 | 古董铺 | 主表，自带完整 CRUD |
| `t_order` | 订单：买家下单、状态流转（待付/已付/取消） | 古董铺·交易 | 主表，自带完整 CRUD |
| `t_order_snapshot` | 订单快照：下单点冻结的商品标题/图/单价/买卖家名 | 古董铺·交易 | **拆分表**（见第三节） |
| `t_file` | 统一图片资产：按 `owner_type+owner_id` 存头像/文物图/商品图 base64 | 资产（共享） | **拆分表**（见第三节） |
| `t_puzzle_score` | 每日词方/填字游戏的成绩与排名 | 填字游戏 | 主表，自带完整 CRUD |

### 模块 → 主表 速查

- 用户：`t_user`
- 文章：`t_article`（收藏 `t_favorite`、点赞 `t_like`）
- 公告：`t_announcement`（已读 `t_announcement_read`）
- 博物馆：`t_relic`
- 古董铺：`t_product`（交易 `t_order` + `t_order_snapshot`）
- 资产：`t_file`（跨模块共享图片）
- 填字游戏：`t_puzzle_score`

## 二、6 张孤儿表（代码已移除，物理表仍在）

这 6 个模块的 **Java 实体/仓库/服务/控制器/DTO 已从代码中删除**，但 JPA `ddl-auto: update` **不会自动 DROP 表**，所以物理表还残留在库里。清理方案见第四节。

| 表名 | 原功能 | 现状 |
|------|--------|------|
| `t_letter` | 读者来信 | 模块移除，待 DROP |
| `t_comment` | 评论 | 模块移除，待 DROP |
| `t_bookmark` | 旧「收藏书签」——**已被 `t_favorite` 取代** | 模块移除，待 DROP（死表） |
| `t_category` | 文章分类 | 模块移除，待 DROP |
| `t_tag` | 文章标签 | 模块移除，待 DROP |
| `t_article_tag` | 文章↔标签 关联表 | 模块移除，待 DROP |

> ⚠️ 注意坑：`t_bookmark`（旧收藏）与 `t_favorite`（现收藏）是两套并存的「收藏」机制，现在只有 `t_favorite` 在用，`t_bookmark` 是死表，可直接删。

## 三、上一轮「表拆分」抽出的两张共享/冻结表

为满足「一表一模块、降低耦合」目标，把内嵌大字段与订单冗余抽成独立表：

1. **`t_file`（图片资产统一表）**
   - 抽出 `t_user.avatar` / `t_relic.image_svg` / `t_product.image` 三个内嵌 base64 大字段。
   - 按 `(owner_type, owner_id)` 唯一约束定位：`AVATAR` / `RELIC_IMAGE` / `PRODUCT_IMAGE`。
   - 实体字段改为 `@Transient`（仅 JSON 出口，字段名 `avatar`/`image` 原样保留），由 `FileAssetService` 读时装配——前端零改动。

2. **`t_order_snapshot`（订单快照表）**
   - 抽出 `t_order` 冗余的商品标题/图片/单价/买卖家名。
   - **下单点冻结写入**，绝不回头 JOIN 商品表——避免改价/下架污染历史订单，守住对账正确性。

幂等迁移：`DbMigrationRunner`（`config` 包）在应用启动时把旧库内嵌 base64 迁到新表，仅当新表无记录时写入，可重复执行。

## 四、孤儿表清理方案

### 方案 A：执行项目已有的 drop 脚本（推荐）

两个脚本已放在项目根目录，包含备份建议与验证查询：

- `drop-bookmark-comment-letter-tables.sql` → 删 `t_letter` / `t_comment` / `t_bookmark`
- `drop-category-tag-tables.sql` → 删 `t_article_tag` / `t_tag` / `t_category`

执行顺序与要点（脚本内已注明）：
- Bookmark/Comment/Letter 三表无真实外键，**可直接 DROP**。
- Category/Tag 三表需**先删从表 `t_article_tag`，再删主表 `t_tag`、`t_category`**，避免外键约束报错。
- 执行前务必先 `mysqldump` 备份（见脚本底部命令）。

### 方案 B：清理前先验证孤儿表是否真存在

在 MySQL 客户端跑（把 `blog` 换成你的库名）：

```sql
SELECT table_name
FROM information_schema.tables
WHERE table_schema = 'blog'
  AND table_name IN (
    't_letter', 't_comment', 't_bookmark',
    't_category', 't_tag', 't_article_tag'
  );
```

- 返回 0 行：这 6 张表已不在库里，无需清理。
- 返回若干行：这些表还在，按方案 A 清理。

清理后复盘：理想状态库里只剩 **12 张活跃表**。

## 五、与「一表一模块 CRUD」目标的对照

活跃表基本达标：

- **主表（自带完整 CRUD）**：`t_user` / `t_article` / `t_relic` / `t_product` / `t_order` / `t_announcement` / `t_puzzle_score` —— 每个对应一个模块。
- **关系/支撑表（依附主表，非耦合违规）**：`t_favorite` `t_like` `t_announcement_read`（用户↔资源关联）、`t_file`（图片资产）、`t_order_snapshot`（订单冻结数据）——有意为之的共享/快照表。

耦合过深未拆的：无（图片与订单快照已在本轮拆出）。
